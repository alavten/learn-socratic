# SoftwareEngineering 提示词真实验证手册（持续执行版）

## 1. 目标（真实提示词导向）

本手册定义可重复执行的“真实提示词会话 + 验收标准”流程，确保：

- 以 `SKILL.md` + `modes/*.md` 为唯一黑盒行为标准；
- 用真实提示词验证 `ingest -> learn -> quiz -> review` 全链路；
- 支持多轮（>=20 轮）与大数据量（全量章节）场景；
- 支持多天（Day1/Day2/Day3）模式切换与图谱反哺闭环；
- 产出可比较、可追踪的持续验证证据。

> 关键原则：本验证用于发现真实用户使用中的问题，不将“固定参数矩阵”或“逐字提示词匹配”作为硬性通过条件。

## 2. 基线与范围

- 资料集：`tests/assert/SoftwareEngineering`（全量章节）
- 方法论基线：`docs/architecture-design.md` 的《1.4 方法论思想概要》
- 核验对象：
  - 意图路由（ingest/learn/quiz/review/shared）
  - 会话契约（summary/next_step）
  - 方法论（SOLO、UBD、Feynman、retrieval-first、spacing-first、元认知）
  - 学习闭环（记录、状态、任务）

## 3. 执行原则（必须遵守）

- 本手册验证的是 **SKILL.md 真实运行行为**，不是测试代码行为。
- 必须以真实提示词驱动会话，按 `SKILL.md` 路由到 `ingest/learn/quiz/review/shared`。
- 允许使用生产 CLI/API 作为数据与状态观测手段；**不允许**用单元测试入口替代真实会话验证。
- 明确禁止以下行为作为“验收通过依据”：
  - 直接调用 `pytest`、`tests/*` 测试函数；
  - 通过 mock/stub 伪造会话结果；
  - 仅检查代码路径覆盖而不进行提示词交互。
- 脚本使用边界（必须遵守）：
  - 允许脚本用于**结果分析**（指标计算、证据对齐、差异对比）；
  - 不允许脚本用于**用例执行**（批量驱动提示词、构造参数化会话输入并据此判定通过）。
- 以下内容不作为硬性通过条件：
  - 强制参数化用例（例如固定 CLI 参数组合矩阵）；
  - 提示词逐字一致（允许语义等价表达）。
- 每条结论必须至少有一条对话证据（prompt/response）或一条生产状态证据（记录/任务/状态快照）。

## 4. 执行频率（持续验证）

- **每次改动 `SKILL.md` / `modes/*.md` 后：** 必跑一次完整验证。
- **每周回归：** 至少跑一次完整验证并对比上次结果。
- **发布前：** 完整跑通并满足全部通过阈值。

## 5. 通过阈值

- 文档覆盖：至少 3 个资料文档进入 learn+quiz+review 闭环（建议 `Chapter4/7/12`）。
- 多轮：总轮次 >= 20（基线）；增强场景 >= 60。
- 跨天：至少 3 天（基线）；增强场景 >= 14 天。
- 方法论命中：
  - learn 命中 Feynman + UBD（Explain/Interpret/Apply）
  - quiz 命中 retrieval-first + SOLO 递进
  - review 命中 spacing-first（overdue/weak/risk）
  - 跨天命中元认知校准（confidence vs result）
- 图谱反哺：至少 1 次“学习缺口 -> 增量 ingest -> 再学习/再测验”成功闭环。

## 6. 提示词套件（推荐基线，允许语义等价改写）

### A. Ingest 启动

1. `我会上传软件工程备考资料，请先帮我导入并建立知识图谱。`
2. `如果资料有格式问题，请按字段告诉我怎么修。`

验收：返回状态明确（success/needs_fix），且有下一步。

### B. Learn（Chapter4）

1. `我们先学计算机网络这一章，请用苏格拉底方式带我理解核心概念。`
2. `我来复述：分层就是把协议随便拆开。`
3. `请用更简单类比重新解释，并问我一个检查题。`

验收：出现 Feynman 闭环（讲解->复述->诊断->简化）。

### C. Learn（Chapter7）

1. `继续学软件工程章节，重点讲需求、设计、测试之间的关系。`
2. `我理解是：只要测试做得好，需求阶段可以弱一点。`
3. `请指出我这个理解哪里错，并让我再复述一次。`

验收：纠错清晰，且保持单概念聚焦。

### D. Learn（Chapter12）

1. `现在切到架构设计章节，帮我建立从质量属性到架构决策的因果链。`
2. `请让我把这个因果链应用到一个电商系统场景。`

验收：从 Explain/Interpret 过渡到 Apply。

### E. Quiz（跨 3 文档）

1. `基于刚才三个章节，开始测验我。一次一题。`
2. `我这题答案是 XXX（故意给一个不完整答案）。`
3. `再来一题更难的。`

验收：retrieval-first，难度按表现升降（SOLO）。

### F. Review（跨 3 文档）

1. `现在做复习，请优先安排我最容易忘的内容。`
2. `这题我不确定，但我猜是 XXX。`
3. `请告诉我下一次复习这三个章节的顺序。`

验收：spacing-first 优先级正确，给出复习窗口建议。

### G. Shared 与切换

1. `继续。`
2. `换成测试模式。`
3. `我其实想先回去补一下讲解。`

验收：先澄清后重路由，切换后契约仍成立。

### H. Day1（建图+学习+测验）

1. `今天是第1天，请先导入这三章资料，然后带我学 Chapter4。`
2. `先讲一个核心概念，我复述后你纠正我。`
3. `现在切到测验模式，按由浅到深出一题。`

### I. Day2（图谱增量完善）

1. `第2天：昨天我发现“质量属性与架构策略”的关系不清楚，请补充到图谱再继续。`
2. `补充后先讲解这个新增关系，再让我用自己的话解释。`
3. `讲完后直接测一题应用题。`

### J. Day3（复习驱动）

1. `第3天：请按最容易忘到最不容易忘安排我复习，覆盖前三天涉及的关键点。`
2. `我先给每题一个把握度百分比，你再看我答得对不对。`
3. `最后给我明天的复习计划，并说明先后顺序原因。`

## 7. 增强用例矩阵（长期 + 大数据量）

### L1. 长期使用场景（14 天）

- 目标：验证长期学习中遗忘风险、任务调度和方法论稳定性。
- 执行要求：
  - 连续 14 天会话；
  - 每天 5-10 轮；
  - 每天至少一次 learn->quiz 或 quiz->review 切换；
  - 每 3-5 轮触发一次元认知校准（信心分 + 错因）。
- 通过阈值：
  - 14 天内无中断性死路由；
  - 复习建议与任务优先级无明显反常（例如长期忽略高风险概念）；
  - 方法论命中率（SOLO/UBD/Feynman/retrieval/spacing）>= 90%。

### L2. 大数据量场景（放大量级）

- 目标：验证在高内容规模下的可用性与稳定性。
- 推荐规模（至少满足一档）：
  - 档 A：200+ concept，800+ relation；
  - 档 B：500+ concept，2000+ relation。
- 执行要求：
  - 先 ingest 全量，再执行 >= 30 轮混合学习会话；
  - 跨 5+ 主题切换；
  - 至少 2 次图谱增量补录。
- 通过阈值：
  - 会话流程可持续推进，无结构性崩溃；
  - 关键字段不缺失（summary/next_step/任务与状态更新）；
  - 随规模增加，方法论行为不明显漂移（命中率下降不超过 10%）。

### L3. 用户噪声场景（真实输入扰动）

- 目标：模拟真实用户不规范输入下的鲁棒性。
- 扰动类型：
  - 口语化与错别字；
  - 半句输入与省略指代；
  - 冲突指令（“先测验但不要出题”）；
  - 跨主题跳转与回退。
- 执行要求：
  - 至少 20 条噪声提示词；
  - 覆盖 shared 澄清与重路由。
- 通过阈值：
  - 澄清问题最小化（避免连续无效追问）；
  - 路由纠正后可恢复主流程；
  - 不出现空转回复。

### L4. 稳定性场景（重复回放）

- 目标：控制同一用例在多次执行中的波动。
- 执行要求：
  - 同一提示词套件完整回放 5 次；
  - 比较每次通过项、失败项、方法论命中率。
- 通过阈值：
  - 硬性指标通过率 100%；
  - 方法论命中率波动 <= 10%；
  - 无新增高优先级退化项。

## 8. 多轮与大数据量执行规则

- 必须加载 `SoftwareEngineering` 全量章节背景。
- 基线轮次 >= 20；增强验证建议 >= 60（learn/quiz/review 尽量均衡）。
- 每天至少 2 次模式切换（例如 learn->quiz->review）。
- 至少一次显式“缺概念/关系冲突”反馈并触发增量 ingest。

## 10. 持续回归判定

- **Pass：** 所有阈值满足，且无阻断性故障。
- **Conditional Pass：** 主阈值满足，但存在轻微偏差（需在下一次回归关闭）。
- **Fail：** 任一硬性阈值未满足（轮次、跨天、方法论核心项、反哺闭环）。

## 11. 复验输出文件建议

每次执行建议在 `docs/acceptance/` 下新增：

- `softwareengineering-prompt-validation-YYYYMMDD.md`（人读报告）
- `softwareengineering-prompt-validation-YYYYMMDD.json`（机器可比对数据）

并在报告中明确“本次相对上次”的变化（通过项、退化项、待修复项）。

## 12. 验收声明模板（报告必须包含）

在每次复验报告结尾追加以下声明：

`本次验收基于 SKILL.md 真实提示词会话执行，未使用单元测试代码或测试函数替代真实交互。所有结论均由对话证据与生产状态证据共同支持。`
