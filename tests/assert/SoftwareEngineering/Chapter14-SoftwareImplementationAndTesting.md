# 第十四章 软件实现与测试

## 一、软件实现

### 1. 程序设计方法

| 方法 | 说明 |
| :-- | :-- |
| **结构化程序设计** | 采用自顶向下、逐步求精的设计方法，各个模块只有一个入口和一个出口。由三种基本结构组成：【顺序】、【分支】、【循环】。 |
| **面向对象的程序设计** | 面向对象的程序设计可大幅度提高软件的可靠性、可维护性和可重用性。 |
| **面向方面的程序设计** | 一种不修改源代码即可动态地为程序统一添加功能的技术。 |
| **可视化程序设计** | 利用编程工具提供的各种组件，像搭积木一样搭建应用程序界面。 |

### 2. 代码重用

（1）**软件重用的三个层次：**【代码重用】、【设计结果重用】（如设计模式或架构重用）、【分析结果重用】（如业务流程重用）。

（2）**代码重用的方式：** 使用函数、对象、模块、类、算法等。

## 二、软件测试

### 1. 软件测试类型

#### 1.1 按程序执行状态分类

**（1）动态测试**【计算机运行程序】

步骤：构造测试用例 → 执行程序 → 分析结果。

**（2）静态测试**【人工监测和计算机辅助分析】

方法：桌面检查、代码走查、代码审查等。

静态测试中的**静态分析**包括：

- **控制流分析：** 检查是否存在未使用的语句、不可达语句、调用了不存在的子程序等。
- **数据流分析：** 检查是否存在对未定义变量的引用、对先前未使用的变量重新赋值等。
- **接口分析：** 检查模块之间、子程序与函数之间接口的一致性，以及形参与实参在个数、顺序、类型上是否一致。
- **表达式分析：** 检查括号是否匹配、数组是否越界、是否出现除数为零等。

#### 1.2 按内部结构的可见性分类

**（1）白盒测试**【结构化测试】

关注程序的内部结构与逻辑。

白盒分析方法包括：**控制流分析**、**数据流分析**、**路径分析**、**程序变异**【错误驱动测试】等。

**路径覆盖**【最强】。

**逻辑覆盖**由强到弱依次为：

**（2）黑盒测试**【功能测试】

关注输入输出及功能。

- **等价类划分：** 不同的等价类能发现不同的问题；包括有效等价类与无效等价类。
- **边界值分析：** 例如条件为 1 ≤ x ≤ 10 时，可取 0、1、10、11 作为测试数据。
- **错误推测：** 依赖测试人员的经验与直觉。
- **判定表：** 最适合描述由多种逻辑条件取值组合形成的复杂情形，并决定应执行的不同动作。
- **因果图：** 根据输入条件与输出结果之间的因果关系设计测试用例。

**（3）灰盒测试**

【既关注逻辑又关注功能】。

#### 1.3 从程序执行的方式分类

**（1）人工测试**

**（2）自动化测试**

流程：编写脚本 → 自动执行。

**不适合的场景：** 项目周期短、需求变更频繁。

**常见自动化测试：** 单元自动化测试、接口自动化测试、UI 自动化测试。

#### 1.4 按测试对象分类

| 测试方法 | 描述 |
| :-- | :-- |
| **功能测试** | 又称**数据驱动测试**。只考虑对各种功能进行测试，不必考虑整个软件的内部结构与代码。 |
| **性能测试** | 包括负载测试、压力测试、并发测试、容量测试、可靠性测试等。负载测试与压力测试通常结合进行。 |
| **安全测试** | 两个层面：应用自身的安全 + 应用所处理数据的安全。 |
| **兼容性测试** | 检验软件之间能否良好协同、是否会相互影响，以及软硬件能否协调高效工作。 |
| **界面测试** | 界面功能模块布局是否合理、整体风格是否一致、命名与样式是否统一、页面是否美观等。 |
| **易用性测试** | 易用性是交互适宜性、功能性与有效性的集中体现，又称用户体验测试。 |
| **稳定性测试** | 通过长时间运行软件，检验是否会出现问题。 |

#### 1.5 新型的测试种类

| 测试方法 | 描述 |
| :-- | :-- |
| **AB 测试（A/B 测试）** | 多个版本同时使用，便于收集各版本的用户反馈并评估最优版本，因此也是一种【网页优化方法】。 |
| **Web 测试** | Web 系统测试与其他系统测试内容基本相同，但测试侧重点不同。Web 代码测试包括：源代码规则分析、链接测试、框架测试、表格测试、图形测试等。 |
| **链接测试** | 可分为三个方面：（1）检验所有链接是否确实指向说明中所指示的页面；（2）检验被链接页面是否存在；（3）保证 Web 应用系统中不存在孤立页面。 |

### 2. 测试阶段划分

| 测试阶段 | 内容 |
| :-- | :-- |
| **单元测试** | 依据详细设计，对模块进行测试，包括模块功能、性能、接口等。 |
| **集成测试** | 依据概要设计，测试模块之间的接口。 |
| **系统测试** | 依据需求文档，在真实环境中验证完整的软件配置项能否与系统正确连接。 |
| **确认测试** | 依据需求文档，验证软件与需求的一致性。包括内部确认测试、Alpha 测试、Beta 测试、验收测试等。 |

**回归测试：** 在软件发生变更后，对变更部分及其是否满足变更后的需求进行正确性验证的测试。

#### 2.1 集成测试策略

**（1）一次性组装方式**

能发现错误，但难以定位原因；故障排查与纠错都会遇到困难。【高风险】

**（2）增量式组装方式**【综合测试】

- **自顶向下的增殖方式：**（需要桩模块）
- **自底向上的增殖方式：**（需要驱动模块）
- **混合增殖方式：**（桩模块与驱动模块都需要）

#### 2.2 系统测试划分

系统测试的常见类型包括：功能测试、性能测试、健壮性测试、用户界面测试、安全性测试、安装与反安装测试等。

**性能测试**还可细分为：

- **负载测试：** 系统在各种工作负载下的性能。
- **压力测试**【上限】：找出系统瓶颈或不可接受的性能点。
- **强度测试**【下限】：在极低资源条件下运行系统。
- **容量测试**【并发测试】：同时在线用户的最大数量。
- **可靠性测试：** MTTF（平均故障间隔时间）等参数。

#### 2.3 系统测试活动与步骤

1. **制定系统测试计划：** 确定人员与任务，明确测试范围、方法、环境与辅助工具。
2. **设计系统测试用例：** 运用等价类划分、边界值分析等测试方法。
3. **执行系统测试：** 执行所设计的测试用例并记录结果。
4. **缺陷管理与纠错：** 排除已发现的错误。

## 三、软件调试

### 1. 软件调试方法

1. **蛮力法：** 主要思想是“靠计算机找错”，效率低、耗时。
2. **回溯法：** 从出错位置沿控制流回溯，直至找到根本原因。程序复杂、路径多时难度较大（适用于小型程序）。
3. **原因排除法：** 主要思想是演绎与归纳，使用二分法实现。

### 2. 软件调试与测试的区别

| 对比方面 | 软件测试 | 软件调试 |
| :-- | :-- | :-- |
| **目标** | 目的是发现已存在的错误。 | 目的是定位并修正错误，使程序得到纠正。 |
| **过程/条件** | 从已知条件出发，按预定过程进行，结果可预知。 | 从未知条件出发；过程的终点不可预知。 |
| **计划性** | 测试过程可以事先设计，进度可以事先确定。 | 调试无法描述过程与持续时间。 |
